SQL & Relational Algebra

SQL Evaluation Order

FROM → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT.

RA Operators

  • σ selection (rows), π projection (cols), ρ rename.
  • ∪ union, − difference, × cross product, ∩ = R−(R−S).
  • ⋈ join = σθ(R×S); natural join projects away duplicate cols.
  • Theta join ⊃ Equi-join ⊃ Natural join.
  • Left anti-join: tuples in R with no match in S. Outer joins: pad unmatched side with NULL.

Equivalences (rewriting heuristics)

  • Push σ down (filter early); push π down (drop attrs early).
  • σ cascade: σp∧q = σp(σq); σ commutes; π cascade.
  • ⋈ assoc/commut; avoid × (replace with ⋈ + condition).

Codd's Theorem

Relational Calculus ≡ Relational Algebra in expressiveness.

Storage & Disk

Hierarchy

CPU Reg (<1ns) → Cache (<10ns) → DRAM (100ns) → SSD (0.1ms) → HDD (10ms) → Network (30ms).

Disk Time

T = Seek + Rot. delay + Transfer

Seq I/O ≫ random (≈100× HDD). Outer tracks have more sectors at same RPM ⇒ higher transfer rate on outer cylinders. Place hot small files on outer tracks, large sequential files in contiguous cylinders.

Page = Block

Unit of I/O, 4–64 KB. Disk-space mgr maps pageId → physical loc.

Prefetching

Reads ahead into pool while CPU works on current page → hides I/O latency on sequential scans. DBMS may prefer its own prefetch (knows access pattern, supports asynchronous I/O, can avoid polluting buffer with read-once pages).

Buffer Management

Pool of frames; each holds pageId, dirty flag, pin counter (not flag — many readers share a page).

API

pin(pid): in pool → pinCnt++; else evict victim (write back if dirty), read in. unpin(pid, dirty): pinCnt−−; OR-in dirty. Also newPage, freePage.

Replacement Policies

  • LRU: evict least-recently unpinned. Bad for sequential flooding (scan with N+1 pages → all misses).
  • Clock: approx LRU; ref bit + circular sweep. Skip pinned; ref=1→clear; ref=0→evict.
  • MRU: better for repeated full scans.

Why pinCount, not pinFlag? multiple operators may need the same page; flag would lose count. Why don't evict on pinCnt=0? Because that page may still be the cheapest kept; replacement chooses based on policy, not first-fit. All dirty? Pick a victim per policy, flush its dirty page (WAL-logs first), then load.

Files, Pages, Records

File Organisations

  • Heap: unordered (linked-list of pages or page directory). Fast insert (2 I/O), slow search.
  • Sorted: keys ordered. Binary search log₂P. Insert costs (log₂P + P).
  • Index: B+ tree (range), hash (equality only).

Page Layout

  • Fixed-length packed: RID = (pid, slot#). Delete moves last record into the gap → RID changes. No good if external pointers exist.
  • Fixed-length unpacked: bitmap of free slots; deletion just clears a bit. RIDs stable.
  • Slotted page: header + slot directory of (offset, length); records grow from end of page toward middle. Variable-length OK; delete sets slot=−1 (RID stable). On overflow, may compact in place.

Slotted vs Packed (exam)

+ stable RIDs, variable-length records, in-place compaction. − per-record indirection (slot dir lookup), tiny space overhead per slot.

Record Layout

Fixed: direct field offset. Variable: array of (offset,len) + NULL bitmap.

Indexes

Index Entries (k*)

A: full record stored in leaf (≤1 such index/table). B: ⟨k, rid⟩. C: ⟨k, [rid list]⟩.

Clustered vs Unclustered

Clustered ≈ data file order matches index, ≤1 per table. Clustered range I/O ≈ #matching pages; unclustered ≈ #matching records (one I/O per RID, worst case).

ISAM

Static tree, overflow chains on insert. Fanout F≫2 → search logF(N). Good for static data (less locking on inner).

B+ Tree

  • Order d: every node has d ≤ #entries ≤ 2d (root may have ≥1).
  • Inner: 2d keys + 2d+1 pointers; leaves: 2d entries + sibling ptr.
  • Always balanced; height = #edges root→leaf.
  • Typical fanout F ≈ 133, fill ~67%. h=3 → ~2.4M; h=4 → ~313M records.
  • Optimisations: prefix compression (inner keys), suffix truncation.
B+ insert (split rule).
  1. Descend to leaf L for key k.
  2. If L has space, insert in sorted order. Done.
  3. If full, split L into L (left) and L' (right). Per exam convention "new right gets larger half": L keeps d entries, L' gets d+1.
  4. Copy L'.firstKey up to parent (leaf split = copy-up).
  5. If parent full, split parent: push middle key up (inner split = push-up, key removed from inner).
  6. Propagate; if root splits, new root with 1 key → height++.
B+ height bounds (insert 1..N, order d).
  • h=0: ≤ 2d entries (single leaf).
  • h=1: 2 ≤ #leaves ≤ 2d+1; entries ∈ [d+1, (2d+1)·2d].
  • h=2: smallest N = (d+1)·d+1 leaves filled to d (min) is too few — for "minimum N to force h=2" use #entries that overflow root at h=1: Nmin,h=2 = 2d·(2d+1)+1
  • Largest N at h=2: full tree = (2d+1)2·2d entries.
  • Min entries in B+ of height h ≥1: (d+1)h·d (root 1 key, every inner d+1 children, every leaf d entries).

Hash Indexes

  • Equality only. h(k) → bucket + chain.
  • Static: chain grows on overflow → degrades.
  • Extendible: directory size 2g (global depth); each bucket has local depth d. Overflow: split bucket; if d=g, double directory.
  • Linear hashing: split next bucket in round even if not the one overflowing → no directory.

External Sorting

N pages, B buffer pages.

Pass 0: ⌈N/B⌉ runs of B pages each (sorted in memory)

Passes 1+: merge B−1 runs at a time → fewer, longer runs

# passes = 1 + ⌈logB−1⌈N/B⌉⌉

Total I/O = 2·N · #passes

Blocked I/O: read/write b pages at a time → fewer seeks but each pass merges ⌊(B−b)/b⌋ runs. Reduces I/O time, may add a pass.

2020 example: N=9600, B=192 → Pass0: 50 runs of 192. Then ⌈log19150⌉=1 pass. Total passes=2, I/O = 2·9600·2 = 38400 page-I/Os. Time = 38400·(10+5+1)ms.

Sort via clustered B+: traverse leaves — strictly better. Unclustered: 1 I/O per record — almost always worse than ext sort.

External Hashing & Aggregation

Phase 1 Partition: hash hp → B−1 partitions, each ≤ B pages ideally.

Phase 2 Rehash: for each partition, build in-mem hash with hr ≠ hp; emit groups/dups.

Need B > √(f·N) (f = fudge factor for in-mem table). Max table B·(B−1) pages.

Use hash for DISTINCT/GROUP BY (scales with #distinct). Use sort if output ordering needed.

  5. Output needs sort → favour SMJ to avoid resort.

BNLJ Tips

Smaller as outer. Reserve 1 page for inner scan, 1 for output, B−2 for outer block. Joining 3 rels at once: outer P pages, two inner each ⌈(N−P)/2⌉ scanned for every outer chunk: cost ≈ N + ⌈N/P⌉·N + ⌈N/P⌉·⌈P/(B−1−P)⌉·N (full inner triple loop). Optimum P trades scans of two inners against rescans of outer.

Sort-Merge

Sort R, S on join key, then walk pointers. Refinement: if Pass-0 runs total ≤ B−1, last merge can also do the join → 3(M+N).

Grace Hash

Phase 1: hash R on hp; hash S on same hp (cost 2(M+N) read+write). Phase 2: per-bucket build (smaller side) + probe (cost M+N). If a bucket overflows, recursive partition with hr.

Outer / Anti-Join Adapts

BNLJ left anti: for each R tuple track "matched" flag; output only unmatched. Same cost as BNLJ. SMJ left anti: merge; if R-key < S-key advance R and emit; if equal skip all R-equal & advance both; if >, advance S. Hash full outer: in build, mark matched bucket entries during probe → at end emit unmatched build entries with NULLs; for probe side emit unmatched probe with NULLs as you go.

Heuristic Rewrites

  • Push σ down past ⋈ when predicate uses one side only.
  • Push π down (drop unused attrs early).
  • Convert × + σ to ⋈.
  • Decompose conjunctions: σp∧q → σp(σq) (separately optimisable).

Access Methods

  • Sequential scan: M I/O.
  • Index scan: composite (A₁..Aₙ) — matches m equality + ≤1 range on next attr (prefix-match rule).
  • Index-only scan: query attrs ⊆ index → no heap I/O.
  • Multi-index / BitmapAnd / BitmapOr: take RID sets from several indexes, intersect/union, fetch heap pages.

EXPLAIN Reading (PG)

Bitmap Heap Scan + BitmapAnd → two indexes (B+ on each cond, must support range). Index-Only Scan → covering index (B+ tree). Hash Join builds hash on inner. Seq Scan → no useful index or full table.

Search Strategy (System R)

  • Left-deep only: inner is base relation → enables pipelining + index NLJ.
  • DP: pass k builds best k-relation plans by extending best (k−1)-plans by 1 relation.
  • Eliminate cross products. Keep cheapest plan per "interesting order" (ORDER BY, GROUP BY, future join keys).
Counting plans (n-relation join).

Transactions & ACID

  • Atomicity: all-or-nothing (logging).
  • Consistency: app + integrity constraints.
  • Isolation: concurrent ≡ serial (CC).
  • Durability: commits survive crashes (WAL).

Schedule S = interleaving of ops preserving per-txn order. Two ops conflict iff same object & ≥1 is W.

ConflictAnomaly
R–WUnrepeatable read
W–RDirty read
W–WLost update

Serializability

Conflict-equiv: same conflict pairs in same order. Conflict-serializable ⟺ ∃ serial conflict-equivalent.

Precedence-graph recipe.
  1. Node per txn.
  2. For each pair of conflicting ops (Ti first, Tj later): add edge Ti→Tj (don't duplicate).
  3. Conflict-serializable ⟺ acyclic.
  4. If acyclic, every topological order is a conflict-equivalent serial schedule. # of such serial schedules = # of topological orderings of the DAG.
  5. If cyclic, remove one op from a cycle to break it; verify by rebuilding graph.

View-serializable: same initial reads, read-from, final writes. View ⊃ Conflict (allows blind writes). NP-hard; not used. Serial ⊂ Conflict ⊂ View ⊂ All.

2PL

Growing (acquire) → Shrinking (release). No new lock after first release. 2PL ⇒ conflict-serializable (but not vice versa).

Strict 2PL

Hold ALL locks until commit/abort. Avoids cascading aborts & dirty reads. Standard in practice.

Deadlocks

Wait-for graph (detection).
  1. Node per active txn.
  2. If Ti waits on a lock held by Tj: edge Ti→Tj.
  3. Cycle ⇒ deadlock. Pick victim (least-work-done / youngest) and abort.

Prevention by timestamps (older = higher priority; restarted txn keeps original ts to prevent starvation).

StrategyOld wants lock held by YoungYoung wants lock held by Old
Wait-DieOld waitsYoung aborts (dies)
Wound-WaitOld preempts (wounds Young)Young waits

Both avoid all deadlocks (no cycle possible) but do abort txns unnecessarily (false positives — txns that wouldn't have deadlocked).

Recovery — WAL & ARIES

FORCEUndo+No-RedoNo-Undo+No-Redo (simplest)

Standard: STEAL + NO-FORCE → must do undo + redo via WAL.

WAL Rules

  • Rule 1: log records for a page flushed before that page → enables UNDO (atomicity).
  • Rule 2: all log records up to commit flushed before commit returns → enables REDO (durability).

Log Record Types

BEGIN, UPDATE(LSN, txn, pid, before, after, prevLSN), COMMIT, ABORT, END, CLR(undoNextLSN), BEGIN_CHECKPOINT, END_CHECKPOINT(ATT, DPT).

Tables

ATT (active txn): {txn → lastLSN, status}. DPT (dirty page): {pid → recLSN = first LSN that dirtied it since last flush}.

ARIES — 3 phases.
  1. Analysis (forward from last BEGIN_CHECKPOINT): rebuild ATT & DPT.
    • UPDATE on pid not in DPT → add (pid, LSN).
    • UPDATE/CLR → ATT[txn].lastLSN = LSN.
    • COMMIT → ATT[txn].status = C; ABORT → status = U.
    • END → remove from ATT.
    • End point: smallest recLSN in DPT (= redoLSN).
  2. Redo (forward from redoLSN): repeat history. For each UPDATE/CLR, if pid ∈ DPT and LSN ≥ recLSN(pid) and pageLSN(pid) < LSN → reapply, set pageLSN=LSN.
  3. Undo (backward): for each loser (status=U in ATT), follow prevLSN chain, write a CLR per UPDATE undone (CLR.undoNextLSN = update.prevLSN). When chain exhausted, write END. CLRs are redo-only (never undone).

Checkpoint

BEGIN_CHECKPOINT — note position. END_CHECKPOINT — write current ATT & DPT to log. Doesn't flush dirty pages (fuzzy). Recovery starts from BEGIN_CHECKPOINT.

Distributed Transactions — 2PC

Lock tables co-partitioned with data. Global deadlock: union waits-for graphs at periodic central node.

2PC protocol (assume all vote yes).
  1. Prepare phase: Coord flush PREPARE (20ms log) → broadcast PREPARE → each Part receives, flush PREPARE → sends YES.
  2. Coord receives all YES → flush COMMIT → broadcasts COMMIT.
  3. Each Part receives COMMIT → flush COMMIT → sends ACK.
  4. Coord receives all ACKs → flush END (often unlogged: presumed-commit).
Timing problem (exam).
  1. Identify slowest Part round-trip: maxi 2·rtt(coord, i).
  2. Total = logcoord-prepare + max{rtt + logpart-prepare + rtt + logcoord-commit... } — work it through participant by participant.
  3. Coord finishes when last ACK arrives + flush END.
  4. Crash of Part during phase 2: coord must wait until Part recovers + redoes commit → ACK. Coord can finish at max(orig finish, recovery time + ack rtt).
  5. Crash of Coord: on recovery reads log; if COMMIT in log → rerun phase 2; if only PREPARE → can presumed-abort or rerun PREPARE depending on protocol.

Conjunctive Queries (CQs)

CQ = {σ, π, ⋈}-fragment = ∃-∧ fragment of FO = simple SELECT-FROM-WHERE with AND + equalities.

Q(x̄) :- R₁(v̄₁), …, Rm(v̄m)

x̄ = head/distinguished vars (must each appear in body).

Semantics — Match / Homomorphism

h: vars(Q) → terms(D), identity on constants, with h(Ri(v̄i)) ∈ D for all i. Output tuple = (h(x̄)). Q(D) = set of all such tuples.

Compute Q(D) by hand.
  1. List atoms; pin every constant.
  2. For one atom at a time, enumerate D-tuples that match the atom's pattern (constants and same-var equalities).
  3. Take the join of partial homomorphisms over shared vars.
  4. Project onto head vars.

Homomorphism Theorem

Q₁ ⊆ Q₂ ⟺ ∃ query hom h: Q₂ → Q₁ (i.e. h(body(Q₂)) ⊆ body(Q₁) and h(head(Q₂)) = head(Q₁)).

UCQ containment: U={Q₁..Qn} ⊆ U'={Q'₁..Q'm} ⟺ ∀i ∃j Qi ⊆ Q'j. (⇐) trivial. (⇒) use canonical DB DQi for each Qi; freeze head as constants; the head tuple is in U(DQi) ⊆ U'(DQi) so some Q'j(DQi) returns it ⇒ hom Q'j→Qi.

Minimisation (compute the core).
  1. For each atom α in body, try to find query hom h: Q → Q\{α} (identity on head vars / constants).
  2. If yes, drop α and repeat from start.
  3. Else keep α.
  4. Result is unique up to var renaming = the core.

Tip: atoms with ≥1 unique variable not appearing elsewhere can never be removed — they witness an existential not derivable from others.

Acyclic CQs & Yannakakis

Hypergraph H(Q): nodes = vars; hyperedges = atoms (set of vars in atom; constants ignored or kept as singletons).

Acyclicity (α-acyclic)

H acyclic ⟺ ∃ join tree T: nodes labelled by hyperedges; for every var v, the set of nodes whose label contains v forms a connected subtree. (a.k.a. running-intersection property)

GYO Reduction (test acyclicity).
  1. Eliminate any vertex contained in ≤1 hyperedge (called an ear tip).
  2. Eliminate any hyperedge that is empty or contained in another.
  3. Repeat until nothing changes.
  4. ∅ ⇒ acyclic (build join tree from elimination order, reverse). Otherwise not acyclic; surviving hypergraph witnesses cyclicity (often a 3-cycle of hyperedges sharing pairwise vars).
Yannakakis (eval acyclic CQ).
  1. Build join tree T; root anywhere.
  2. Up-pass: for each leaf-to-root edge, semi-join child into parent: parent ← parent ⋉ child.
  3. If root empty → answer ∅ (no match). Else for boolean CQ → YES.
  4. Down-pass (for full answer): semi-join parent into child top-down → all relations now contain only "useful" tuples.
  5. Compute the join along the tree; project onto head vars.

Acyclic CQ evaluation: O(‖D‖·‖Q‖) — linear time. Bounded treewidth / hypertree-width generalises to tractable cyclic classes.

Parallelism

  • Inter-query: throughput.
  • Intra-query inter-op: pipeline + bushy.
  • Intra-query intra-op: data partition + per-partition op.

Partitioning

  • Round-robin: even load, no locality.
  • Hash(A): equality lookup on A, equi-join on A, group-by A — local.
  • Range(A): range queries + above.
Distributed query plan (decision tree).
  1. Selection on partition key → only relevant nodes; else broadcast or all-nodes.
  2. GROUP BY g — if partitioned on g: local agg, no shuffle. Else: local partial agg → reshuffle by g → final agg.
  3. MAX/SUM/COUNT: distributive → local + final aggregate.
  4. Join R⋈AS: see scenarios below.

Distributed Join Scenarios

  1. One side replicated everywhere → local join.
  2. Both partitioned on join key → local join.
  3. Different keys, one side small → broadcast small side.
  4. Different keys, both large → reshuffle both on join key.

Parallel Hash Join Cost

Round-robin start, k machines, R+S total: shuffle reads/writes = 2(M+N)/k per machine; rebuild = (M+N)/k. Total cross-network ≈ (M+N)·page_size·(k−1)/k. With combined partition+phase1: 3(M+N)/k disk I/Os per machine.

Parallel SMJ (range-partitioned same)

Local sort + merge: per machine 3·(M+N)/k if refined SMJ applies.

NoSQL

Web-scale needs horizontal scale; 2PC blocking; strong consistency expensive.

Optimistic CC & Hashing Recap

Optimistic CC (3 phases)

Read (private workspace) → Validate (check no conflict with concurrently-committed) → Write. No locks during read. Aborts costly under contention; great when conflicts rare.

View ⊋ Conflict — example

S = W₁(A) W₂(A) W₃(A) — all blind writes on A. S is view-equiv to T₁T₂T₃ but adding any reads can break conflict-equiv. View-serializable but not conflict-serializable.

Static-hash search costs (skewed function)

Uniform hp over [0,B−1]: each bucket = N/B records, search = ⌈N/(B·R)⌉ I/Os always.
Skewed hd(k) = #digits(k): bucket sizes = digit-count distribution. Min I/O = bucket with fewest keys; Max I/O = largest bucket; Expected = Σ p(bucket) · I/O(bucket).
