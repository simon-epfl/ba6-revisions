1. SQL & Relational Algebra

SQL Evaluation Order

FROM → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT

RA Operators

  • σ Selection (rows), π Projection (cols), ρ Rename
  • ∪ Union, – Difference, × Cross Product
  • ∩ Intersection = R – (R – S)
  • ⋈ Join = σθ(R × S); Natural join projects away dups
  • Theta join ⊃ Equi-join ⊃ Natural join

Codd's Theorem

Relational Calculus = Relational Algebra in expressiveness.

2. DBMS Architecture

Layers (bottom→top): Disk Space Mgmt → Buffer Mgmt → Files & Index Mgmt → Operator Execution → Query Planning. Cross-cutting: Recovery, Concurrency Control.

3. Storage & Disk

Storage Hierarchy

CPU Reg(<1ns) → Cache(<10ns) → DRAM(100ns) → SSD(0.1ms) → HDD(10ms) → Network(30ms)

Disk Access Time

Time = Seek + Rotational delay + Transfer

Sequential I/O ≫ Random I/O (100× on HDD). SSD: random reads ~500MB/s, seq ~525MB/s; random writes ~120MB/s.

Page = Block = unit of I/O

Typical 4–64KB. Disk space mgr: maps page IDs → physical locations.

4. Buffer Management

Buffer pool = in-memory cache of pages. Each frame: PageId, Dirty flag, Pin count.

Pin/Unpin

pin(pageno): if in pool, pinCount++; else evict victim (if dirty→write back), read page. unpin(pageno, dirty): pinCount--, set dirty.

Replacement Policies

  • LRU: evict least recently unpinned. Good for random access, bad for sequential flooding (scan with N+1 pages → all misses).
  • Clock: approx LRU. Circular buffer + ref bit. Skip pinned; if ref=1→clear; if ref=0→evict.
  • MRU: evict most recently used. Better for repeated scans.

5. Files, Pages, Records

File Organisations

  • Heap: unordered. Linked list or page directory.
  • Sorted: sorted by key. Binary search: log₂P.
  • Index: B+ tree, hash.

Page Layout

  • Fixed-length packed: RID = (pageId, position). Delete → move last record (changes RID).
  • Fixed-length unpacked (bitmap): gaps OK, bitmap tracks free slots.
  • Slotted pages: slot directory → offsets. Supports variable-length. RID = (pageId, slotId). Records stored from end. Delete → set slot to -1.

Record Layout

Fixed: direct field access via arithmetic. Variable: array of (offset, length) pairs; NULL bitmap.

8. Indexes

Index Entries (k*)

A: record stored in index (at most 1 per table). B: <k, rid>. C: <k, [rid list]>.

Clustered vs Unclustered

Clustered: data records ≈ same order as index. At most 1 clustered index/table. Clustered I/O = #matching pages. Unclustered I/O ≈ #matching records.

ISAM

Static tree. Overflow chains on insert. No locking on non-leaf. Good for static data. Search: logF(N). F≫2 so much better than binary search.

B+ Tree

  • Always balanced, dynamic. Each node d ≤ entries ≤ 2d (except root).
  • Typical order ~100, fill factor ~67%, fanout F≈133.
  • Height 3 → ~2.4M records, height 4 → ~313M records.
  • Insert: find leaf; if full, split (copy up middle key for leaf, push up for inner). May propagate to root.
  • Delete: remove entry; if underflow, redistribute or merge. In practice, just leave space.
  • Optimisations: prefix compression, suffix truncation.

Hash Index

  • Equality only (no range). h(k) → bucket.
  • Static chaining: overflow chains degrade perf.
  • Extendible hashing: directory of pointers. Global depth n, local depth d. Split bucket on overflow; double directory if local=global.

9. External Sorting

N pages, B buffer pages.

Pass 0: ⌈N/B⌉ sorted runs of B pages

Passes 1+: merge B-1 runs at a time

# passes = 1 + ⌈logB-1(⌈N/B⌉)⌉

Total I/O = 2N × (# passes)

Sorting via clustered B+ tree: always better. Unclustered: almost always worse (1 I/O per record).

10. External Hashing & Aggregation

Phase 1 (Partition): hash hp into B-1 partitions. Phase 2 (Rehash): build in-memory hash table per partition with hr ≠ hp.

Max table size: B·(B-1). Need B > √(f·N).

Hashing preferred for DISTINCT/GROUP BY (scales with #distinct values). Sort preferred if output needs ordering.

13. Query Optimisation

Pipeline: SQL → Parser → Rewriter → Optimiser → Executor

Plan Space

  • RA equivalences: σ cascade/commute, π cascade, × assoc/commut, ⋈ assoc/commut (beware cross-products).
  • Heuristics: push selections down, push projections down, avoid cross-products, break complex predicates.
  • Physical choices: heap scan vs index scan; NLJ vs SMJ vs HJ.

Access Methods

  • Sequential scan: I/O = N pages.
  • Index scan: composite key (A₁,…,Aₙ) matches m equality + ≤1 range on Am+1.
  • Index-only scan: answer from index alone (no data pages).
  • Multi-index/Bitmap scan: intersect/union RID sets from multiple indexes.

15. Search Strategy (System R)

  • Left-deep trees only: inner always base relation → enables pipelining & index NLJ.
  • Dynamic programming: Pass k builds best k-relation plans from (k-1)-relation plans + 1 relation.
  • Eliminate cross-products; retain cheapest plan + cheapest per interesting order (ORDER BY, GROUP BY, future join keys).

16. Transactions & ACID

  • Atomicity: all or nothing (logging / shadow paging).
  • Consistency: app responsibility; DB integrity constraints.
  • Isolation: concurrent txns appear serial (CC protocols).
  • Durability: committed changes survive crashes (WAL).

Formal Model

Txn T = ⟨R(A), W(A), …⟩. Schedule S = interleaving of all txn operations preserving per-txn order.

17. Serializability

Conflict Serializability

S is conflict serializable iff S can be transformed into a serial schedule by swapping consecutive non-conflicting ops.

Dependency (Precedence) Graph: node per txn, edge Ti→Tj if conflicting ops and Ti's op is first. Conflict serializable ⟺ acyclic graph. Topological sort gives equivalent serial order.

View Serializability

Weaker: same initial reads, same read-from, same final writes. Allows "blind writes". NP-hard to check. No DBMS uses it.

Serial ⊂ Conflict Serializable ⊂ View Serializable ⊂ All Schedules.

18. Locking & 2PL

Lock Types

S-LockX-Lock
S-Lock✓✗
X-Lock✗✗

Two-Phase Locking (2PL)

Growing phase: acquire locks. Shrinking phase: release locks. Cannot acquire after first release. 2PL guarantees conflict serializability.

Strict 2PL

Hold ALL locks until txn commits/aborts. Prevents cascading aborts & dirty reads. Most common in practice.

Deadlocks

  • Detection: waits-for graph; cycle = deadlock. Abort a "victim" to break cycle.
  • Prevention: priorities by timestamp (older = higher).
    Wait-Die: old waits for young; young aborts.
    Wound-Wait: old preempts young; young waits for old.
  • Restarted txn keeps original timestamp (prevents starvation).

19. Recovery

Buffer Pool Policies

STEALNO-STEAL
FORCE—No undo, No redo (simplest)
NO-FORCEUndo + Redo (fastest runtime)—

Almost all DBMS: STEAL + NO-FORCE (best runtime perf, requires undo+redo).

Write-Ahead Logging (WAL)

  • Log record: <TxnId, Object, OldVal, NewVal>
  • Rule 1: all log records for a page flushed to disk before the page is written to disk (supports UNDO → atomicity).
  • Rule 2: all log records flushed before txn commit returns (supports REDO → durability).

ARIES Recovery (3 phases)

  1. Analysis: scan WAL from last checkpoint → identify dirty pages & active txns at crash.
  2. Redo: replay ALL actions from appropriate log point (repeat history).
  3. Undo: reverse uncommitted txns.

20. Distributed Transactions

Distributed Locking

Lock tables partitioned with data. Global deadlock: union waits-for graphs periodically at designated node.

Two-Phase Commit (2PC)

Phase 1 (Vote): Coordinator sends PREPARE. Participants log PREPARE*, vote Yes/No. Must be unanimous.

Phase 2 (Commit): Coordinator logs COMMIT*, broadcasts. Participants log COMMIT*, send ACK. Coordinator logs END.

* = flush log before sending next message.

Recovery with 2PC

2PC is unavailable during any single failure (blocking). Paxos Commit addresses this.

21. Conjunctive Queries (CQs)

CQ = {σ,π,⋈}-fragment of RA = ∃,∧ fragment of relational calculus = simple SELECT-FROM-WHERE (AND + equality only).

Rule form: Q(x) :- R₁(v₁), …, Rm(vm)

Semantics

A match/homomorphism h maps query vars → DB values (identity on constants), s.t. h(body) ⊆ D. Answer = {(h(x₁),…,h(xk)) | h is a match}.

Complexity

ProblemComplexity
CQ-Evaluation (combined)NP-complete
CQ-Evaluation (data)PTIME (||D||O(||Q||))
CQ-SatisfiabilityTrivial (always yes via canonical DB)
CQ-ContainmentNP-complete
CQ-EquivalenceNP-complete

Homomorphism Theorem

Q₁ ⊆ Q₂ ⟺ ∃ query homomorphism from Q₂ to Q₁.

Minimisation

Remove redundant atoms: if ∃ query hom from Q :- body to Q :- body∖{α}, remove α. Result is unique (up to renaming) = core.

22. Acyclic CQs & Fast Evaluation

Hypergraph of CQ

Nodes = variables, hyperedges = sets of variables in each atom.

Acyclicity

Hypergraph is acyclic iff it has a join tree: tree T with labels L(n) = hyperedge, s.t. for each variable v, nodes containing v form connected subtree.

GYO Reduction

  1. Eliminate nodes in ≤1 hyperedge.
  2. Eliminate empty/contained hyperedges.
  3. Repeat. Acyclic ⟺ GYO reduces to ∅. O(||Q||) time.

Yannakaki's Algorithm

  1. Compute join tree of H(Q).
  2. Assign corresponding relations to nodes.
  3. Semi-joins bottom-up.
  4. Root non-empty → YES.

Acyclic CQ evaluation: O(||D||·||Q||) — linear time!

Bounded treewidth (graph) / hypertree width (hypergraph) → tractable classes.

23. Parallel & Distributed DBMS

Architectures

TypeSharesPros/Cons
Shared MemoryMem+DiskEasy SW, costly HW, limited scale
Shared DiskDiskMiddle ground, easy consistency
Shared NothingNothingCheap, scales well, harder SW

Parallelism Types

  • Inter-query: different queries in parallel → throughput.
  • Intra-query, inter-operator: pipeline & bushy parallelism.
  • Intra-query, intra-operator: partition data, run op on each.

Partitioning

  • Round-robin: good for load spreading.
  • Hash: good for equality lookups, equi-joins, group-by.
  • Range: good for range queries + above.

Parallel Operators

Hash join: hash-partition both on join key → local hash join. Sort-merge: range-partition both → local SMJ. Scan: scan in parallel, merge.

Distributed Join Scenarios

  1. One table replicated → local join.
  2. Both partitioned on join key → local join.
  3. Different keys, one small → broadcast small table.
  4. Different keys, both large → reshuffle both.

Query planning: must consider network cost, data location, skew/bottlenecks.

24. NoSQL

Motivation

Web-scale apps need horizontal scaling. 2PC too slow. Distributed RDBMS consistency too expensive.

CAP Theorem

Consistency (all nodes see same data) + Availability + Partition tolerance — pick 2 of 3.

BASE

Basically Available, Soft state, Eventually consistent.

NoSQL Types
